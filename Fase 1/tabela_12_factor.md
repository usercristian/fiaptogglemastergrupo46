# Análise 12-Factor App - Fase 1

Esta tabela detalha a conformidade do projeto com os princípios do 12-Factor App na fase atual.

| Fator | Descrição | O fator é atendido pelo projeto em sua fase atual? | Justificativa |
|:---|:---|:---|:---|
| I. Base de Código | Uma base de código com rastreamento utilizando controle de revisão, muitos deploys | Sim | O projeto utiliza um respositório único para gerar o deploy tanto no docker local quanto no ambiente EC2 da AWS. Em ambos os casos é possivel o rastreamento e controle de versões através do uso do GIT. |
| II. Dependências | Declare e isole as dependências | Não | O arquivo requirements.txt lista os pacotes mas não "trava" as versões exatas de todas as sub dependências. Como exemplo a library setuptools não possuia uma versão especifica estabelecida como requisito, o que fez necessária a sua instalação durante a implementação. O instalador, por padrão, baixa e executa a versão mais recente. Se o mesmo projeto for instalado em situação posterior, pode ocorrer que uma dessas sub dependências tenha sido atualizado "quebrando" o pacote gerando incompatibilidades e falhas de execução. |
| III. Configurações | Armazene as configurações no ambiente | Sim | Como exemplo, o projeto se utiliza de variáveis de ambiente como o DB_HOST e DB_PASSWORD para se conectar ao banco. |
| IV. Serviços de Apoio | Trate os serviços de apoio, como recursos ligados | Não | O serviço de banco de dados é tratado como um recurso externo conectável, cuja configuração é realizada por meio de variáveis de ambiente, permitindo flexibilidade na substituição do serviço sem alterações no código da aplicação. Não há serviços ou scripts de apoio a aplicação configurados nesta fase. Um exemplo é que não existem serviços de validação ou ainda reconexão automática caso ocorra uma queda de conexão com o banco de dados. |
| V. Construa, lance, execute | Separe estritamente os build, release e run em estágios | Não | Atualmente, o processo de deploy é manual. De acordo com as melhores práticas aqui utilizadas como referência, a criação de imagens deve estar apartada das demais fases da implantação. |
| VI. Processos | Execute a aplicação como um ou mais processos que não armazenam estado | Não | O código depende de arquivos e variáveis temporárias. As melhores práticas indicam a necessidade de implantação dessas variáveis de forma independente. |
| VII. Vínculo de porta | Exporte serviços por ligação de porta | Sim | A aplicação é independente e expõe o serviço diretamente através da porta 5000. |
| VIII. Concorrência | Dimensione por um modelo de processo | Não | Não há balanceamento de carga configurado nesta etapa. |
| IX. Descartabilidade | Maximizar a robustez com inicialização e desligamento rápido | Não | O código não trata sinais de desligamento e não finaliza tarefas já for a de uso de forma automatizada. |
| X. Dev/prod semelhantes | Mantenha o desenvolvimento, teste, produção o mais semelhante possível | Não | Não há ambientes de produção, desenvolvimento ou homologação separados nesta fase. Todo o código é executado em um ambiente único. |
| XI. Logs | Trate logs como fluxo de eventos | Sim | A aplicação exporta informações de log permitindo a coleta por ferramentas de monitoramento externas. |
| XII. Processos de Admin | Executar tarefas de administração/gerenciamento como processos pontuais | Não | Não existem, por exemplo, scripts que automatizem um possível processo de migração de base de dados implantado nessa fase. |
